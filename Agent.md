# Agent

## 架构
- 执行框架
ReAct / Plan-and-Execute / AdaPlanner
- 推理结构
CoT / ToT / GoT
- 记忆系统
ReAct with Memory
- 自我优化
Reflexion / CoVe

## 执行框架
### ReAct
- 特点：边思考边行动，按“思考 → 执行 → 观察 → 再思考”的循环推进。
- 优点：简单直接，容易落地；能结合实时反馈逐步修正；比纯 CoT 更可执行；过程更可解释。
- 缺点：容易短视和局部最优；复杂探索任务里可能不够“深”；计划质量强依赖上下文和工具观察质量。
- 适用场景：需要边查边做、信息不完整、依赖外部工具的任务，如检索、问答、排错、网页/数据库操作。

### Plan & Execute
- 特点：先生成初始计划，再按步骤执行；规划和执行相对分离；必要时根据执行反馈重规划。
- 优点：结构清晰，便于控制任务流程；比 ReAct 更强调全局路线；适合中等复杂度的多步骤任务。
- 缺点：初始计划质量很关键；对环境变化的反应通常不如 ReAct 灵活；如果初始规划偏差大，后续可能需要多次修正。
- 适用场景：目标较明确、步骤可拆解、需要先有整体路线图的任务，如报告写作、项目拆分、批量流程执行。

### AdaPlanner
- 特点：强调自适应规划，执行过程中持续感知状态并动态调整后续计划；比传统 Plan & Execute 更频繁地重规划。
- 优点：对复杂、动态、不确定任务更稳；能更主动地利用新反馈修正路径；不容易被早期错误计划绑定。
- 缺点：系统更复杂；规划与重规划开销更高；对状态表示、反馈质量和决策机制要求更高。
- 适用场景：环境变化快、信息不完整、任务路径不确定性高的场景，如复杂调研、动态工具任务、长链路任务编排。

### Plan & Execute vs AdaPlanner
- 相同点：都属于“先规划、后执行”的范式，不是像 ReAct 那样一步一动。
- 不同点：
  - Plan & Execute 更像先定好路线，执行中出问题再改。
  - AdaPlanner 更像执行过程中一直看路况，随时可能改路线。
- 实现上的差别：
  - Plan & Execute 往往把计划表示成较固定的步骤列表，执行器按列表推进。
  - AdaPlanner 往往把计划看成可变的状态结构，会根据执行反馈持续更新后续步骤。
  - Plan & Execute 的重规划更偏事件触发；AdaPlanner 的重规划更偏持续感知和在线调整。
- 司机类比：
  - Plan & Execute：像司机出发前先定路线，路上如果堵车或改道受阻，再临时换路线。
  - AdaPlanner：像导航软件，开车时一直看实时路况，发现拥堵、限行、天气变化就马上重算路线。
  - 直观总结：Plan & Execute 是“先定好再做，出问题再改”；AdaPlanner 是“边做边看，随时都可能改”。


## 推理结构
### CoT
- 特点：线性逐步推理，适合把任务从目标顺着展开成步骤计划。
- 生成 plan 的方式：通常通过 prompt + few-shot 直接让模型输出分步计划，工程上最简单。
- 控制点：主要依赖提示词来约束步数、粒度和输出格式。
- 适用场景：目标清晰、步骤较稳定、希望快速得到一个可执行初始计划的任务。

### ToT
- 特点：树状多分支推理，先生成多个候选，再评估和剪枝，保留更优分支继续展开。
- 生成 plan 的方式：不是只写一条线，而是先产生多个候选计划或子计划，再通过搜索选出更优方案。
- 控制点：通常需要程序显式控制搜索过程，例如 BFS、DFS、beam search、评分和剪枝。
- 适用场景：方案空间较大、初始计划质量很关键、需要比较多个路径的任务。

### GoT
- 特点：图结构推理，允许节点复用、路径交叉和依赖共享，比 ToT 更结构化。
- 生成 plan 的方式：把任务拆成节点，并维护节点之间的依赖关系，允许中间结论被多个后续步骤复用。
- 控制点：通常也需要程序管理图结构、节点扩展、合并、回溯和依赖关系。
- 适用场景：子任务之间有明显依赖关系、存在结果复用、需要更强结构化管理的复杂任务。

### 在 Plan & Execute 中怎么用这三者
- CoT：适合直接生成一个初始计划，作为最轻量的规划器。
- ToT：适合在规划阶段搜索多个候选计划，再选出更优方案执行。
- GoT：适合把任务组织成有依赖关系的结构图，便于复用中间结论并管理复杂任务。
- 总结：CoT 更偏“顺着写计划”，ToT 更偏“搜索多个计划”，GoT 更偏“结构化组织计划”。

## 自我优化
### Reflexion
- 核心：更像 bad case 收集与复盘机制。先经历一次失败或低质量输出，再总结“哪里错了、为什么错、下次怎么改”。
- 触发时机：通常在一次任务或一次完整尝试结束后，再根据任务结果、环境反馈、工具执行结果、标准答案或评分器结果判断是否失败。
- 实现方式：
  - 人工复盘：人工分析 bad case，修改 prompt、上下文模板、流程或工具调用顺序。
  - 自动反思：用 LLM 自评、评分器、环境反馈或工具指标判断失败，再生成反思记忆并写入后续流程。
- 特点：偏“从失败中学习”，强调把失败样本转化为可复用经验；更适合做持续优化，而不是即时验收。
- 适用场景：重复任务多、希望随着运行逐步提升、需要积累经验的场景。

### CoVe（Chain of Verification）
- 核心：更像答案验证链。先生成答案或中间结论，再沿着一条验证链对关键点进行系统检查，发现问题后修正。
- 触发时机：通常在最终输出前进行验证；也可以在中间结果或关键节点后做检查，但不要求每一步都检验。
- 实现方式：
  - LLM 作为 verifier：让模型针对关键断言生成验证问题，再逐项核验并修正。
  - 工具/规则验证：用程序、单测、正则、schema、计算器或外部检索核实容易出错的内容。
  - 评分器/Judge：用标准答案、评价模型或任务指标判断当前输出是否可靠。
  - 人工复核：一般用于高风险场景或抽样审核，不是 CoVe 的核心依赖。
- 特点：偏“先答后查”，重点是提高当前输出的正确性和一致性；更像质检或审稿，而不是积累长期经验。
- 适用场景：对事实、逻辑、计算或格式正确性要求高的任务。

### Reflexion vs CoVe
- Reflexion：偏向“事后反思，收集 bad case，积累经验，改进下次表现”。
- CoVe：偏向“先答后查，对当前答案做验证和修补”。
- 一句话记忆：Reflexion 是“做完后复盘，下次别再犯”；CoVe 是“先答再查，看看有没有错”。
